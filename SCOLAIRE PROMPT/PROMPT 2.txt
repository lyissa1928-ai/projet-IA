1.3 Endpoints Auth (à implémenter)
POST /api/v1/auth/login

Body:

{ "matricule": "STRING", "password": "STRING" }

Réponse:

{
  "accessToken": "...",
  "refreshToken": "...",
  "user": { "id": "...", "matricule": "...", "role": "...", "firstLoginRequired": true }
}

Règles :

Si status ≠ ACTIVE → refuser (ACCOUNT_DISABLED)

Password hash compare

Mise à jour lastLoginAt

Audit log : AUTH_LOGIN_SUCCESS (et AUTH_LOGIN_FAILED en cas d’échec, sans exposer la cause exacte)

POST /api/v1/auth/refresh

Body:

{ "refreshToken": "..." }

Réponse:

nouveaux tokens (rotation refresh) + user

POST /api/v1/auth/logout

Révoque le refresh token courant (ou tous si option)

POST /api/v1/auth/change-password

Protégé JWT.
Body:

{ "oldPassword": "...", "newPassword": "...", "confirmNewPassword": "..." }

Règles :

Politique mot de passe (min 8 + maj + min + chiffre + spécial)

Si firstLoginRequired=true, obligatoire de passer par change-password

Mettre firstLoginRequired=false après succès

Audit : AUTH_PASSWORD_CHANGED

POST /api/v1/auth/force-first-login-change

Optionnel : endpoint interne (admin) pour rebasculer firstLoginRequired=true.

1.4 Guards & RBAC
1.4.1 JwtAuthGuard

Vérifie access token

Injecte req.user

1.4.2 RolesGuard + decorator @Roles(...)

Empêche accès selon rôle

1.4.3 “FirstLoginGuard”

Pour toutes les routes /app/** côté frontend on gère UX, mais côté backend :

Pour endpoints sensibles (ex: user profile update), refuser si firstLoginRequired=true sauf /auth/change-password

1.5 Sécurité

Rate limiting login (par IP + matricule)

Messages d’erreur non révélateurs

Password hashing Argon2 (ou bcrypt cost élevé)

Refresh token stocké hashé en DB

Rotation refresh token (nouveau à chaque refresh)

CORS strict (origins du frontend)

Helmet + secure headers
